DB Migration

Ncloud(네이버 클라우드) VPC환경에서 Rocky Linux 서버에 설치형 PostgreSQL DB를 설치하고, 접속하는 방법입니다.

ToC

개요

네이버 클라우드 환경으로 데이터베이스를 이관하기 위해 온 프레미스(On-Premise) 및 기타 다양한 환경에서 활용 가능한 Data Migration Service를 활용하여 네이버 클라우드의 Cloud DB for MySQL로 데이터베이스 마이그레이션 테스트를 진행하고자 합니다.

사전구성

Source DB: 네이버 클라우드 Centos 7.8 서버(hostname: testdb)를 생성하고 Maria DB를 설치합니다.

Target DB : 네이버 클라우드 Cloud DB for MySQL 서버를 생성합니다.

※ Source DB는 네이버 클라우드 외부에 위치한 경우가 많을 것으로 예상하여 두 DB 간 공인 IP 통신을 기본으로 작성하도록 하겠습니다.

DB 버전 정보

Source DB: MariaDB 10.6.15 (NCP/VM설치형)

Target DB: MySQL 8.0.32 (NCP/Cloud DB for MySQL)

※ Cloud DB for Mysql로 이전하는 경우 Source DB로 Mysql, MariaDB 모두 지원 가능하며 버전별 설정상 약간의 차이점이 존재할 뿐 대부분은 동일합니다. 상세한 내용은 아래에서 자세히 다루도록 하겠습니다.

Target DB 작업 리스트

1. SourceDB의 공인 IP 및 접속 포트 확인

  • Source DB의 VM을 Private Subnet에 생성한 경우에는 NatGateway의 공인 IP를 확인합니다.
  • Source DB로 생성한 testdb의 공인 IP 정보를 확인합니다.

2. Target DB의 ACG에서 SourceDB의 IP를 접근 소스로 Outbound 등록 설정

※ 마이그레이션은 Target DB가 Source DB에 접근해서 데이터를 가져오는 방식으로 동작합니다. 따라서 Target DB에는 Source DB의 IP로 Outbound 오픈이 필요하며, Source DB는 Target DB가 접근할 수 있도록 Target DB의 IP를 Inbound에 열어줘야 합니다.

Definer 계정 확인

SELECT TABLE_SCHEMA, TABLE_NAME FROM INFORMATION_SCHEMA.COLUMNS   WHERE COLUMN_NAME = 'DEFINER';

SELECT DISTINCT DEFINER FROM INFORMATION_SCHEMA.EVENTS;
SELECT DISTINCT DEFINER FROM INFORMATION_SCHEMA.ROUTINES;
SELECT DISTINCT DEFINER FROM INFORMATION_SCHEMA.TRIGGERS;
SELECT DISTINCT DEFINER FROM INFORMATION_SCHEMA.VIEWS;

3. Target DB에 Source DB의 Definer 계정 생성

  • 콘솔에서 SourceDB에 생성된 Definer 유저명과 접속 호스트 정보로 동일한 유저 생성 후 DDL 권한을 부여합니다.
Source DB에 생성된 유저의 접근 호스트가 ‘localhost’ 또는 ‘127.0.0.1일 경우 마이그레이션 전에 ‘%’로 변경 후 마이그레이션을 진행하거나 마이그래이션 후 ‘%’로 변경하길 네이버 클라우드는 권고하고 있습니다.
  • 계정이 “abc”같이 세 자리 계정은 생성 가능한 유저의 최소 글자 수가 4자로 제한되어있는 문제로 콘솔에서는 생성이 불가하기 때문에, 직접 DB에 접속하여 생성하고 아래와 같이 권한을 부여합니다.
CREATE USER 'abc'@'%' IDENTIFIED WITH 'mysql_native_password' BY 'Test123$';

GRANT PROCESS, SHOW DATABASES, REPLICATION SLAVE, REPLICATION CLIENT, CREATE USER ON *.* TO `abc`@`%` WITH GRANT OPTION;
GRANT SHOW_ROUTINE ON *.* TO 'abc'@'%' WITH GRANT OPTION;
GRANT SELECT ON `mysql`.* TO `abc`@`%` WITH GRANT OPTION; 
GRANT SELECT, EXECUTE ON `sys`.* TO `abc`@`%` WITH GRANT OPTION;                                                         
GRANT ALL PRIVILEGES ON `testdb`.* TO `abc`@`%` WITH GRANT OPTION;                                                       
GRANT SELECT ON `performance_schema`.* TO `abc`@`%` WITH GRANT OPTION;                                                   
GRANT INSERT, UPDATE, DELETE ON `mysql`.`time_zone_leap_second` TO `abc`@`%`;                                            
GRANT INSERT, UPDATE, DELETE ON `mysql`.`time_zone_name` TO `abc`@`%`;                                                   
GRANT INSERT, UPDATE, DELETE ON `mysql`.`time_zone_transition_type` TO `abc`@`%`;                                        
GRANT INSERT, UPDATE, DELETE ON `mysql`.`time_zone_transition` TO `abc`@`%`;                                             
GRANT INSERT, UPDATE, DELETE ON `mysql`.`time_zone` TO `abc`@`%`;

4. Target DB 데이터베이스 확인

  • Source DB에 생성된 데이터베이스 명과 동일한 데이터베이스가 Target DB에 존재해서는 안 됩니다. 만약 동일한 데이터베이스가 존재한다면 Target DB에 생성된 데이터베이스를 반드시 삭제합니다.

Source DB 작업 리스트

1. 공인 IP 및 접속 포트 확인

  • Target DB가 Private Subnet에 생성한 경우 NatGateway의 공인 IP를 확인합니다.
  • Public Subnet에 생성한 경우 Public 도메인 부여 후 nslookup으로 공인 IP를 확인합니다.
    • Public 도메인 부여

    • Public 도메인 확인

2. Target DB의 공인 IP를 접근 소스로 하여 Inbound에 등록합니다.

3. Source DB에 마이그레이션 전용 유저 생성

  • 패스워드는 최소 길이 8자 이상, 특수문자 1개 이상, 숫자 1개 이상, 대소문자 조합 1개 이상 필요합니다.
  • 허용 호스트 설정 부분에서는 마이그레이션 실행 시 Targer DB가 Source DB로 접근이 가능하도록 해야 합니다.
  • 기본DB 제외 유저가 추가로 생성한 모든 데이터베이스에 대해 아래 권한을 부여합니다.
  • 아래 예시를 참고하여 마이그레이션 계정을 생성합니다.

GRANT SELECT ON mysql.* TO 'migration_test'@'접근호스트';

GRANT SELECT, SHOW VIEW, TRIGGER ON [사용자 DB].* TO 'migration_test'@'접근호스트';

FLUSH PRIVILEGES;

4. Source DB의 Character Set/Collation 점검 및 설정 변경

  • Cloud DB for MySQL은 [utf8, utf8mb4, euckr] Character Set에 대해서만 지원합니다. 아래 점검 쿼리 결과에 따라 지원 가능한 캐릭터셋으로 변경 후 진행해 주시기 바랍니다.
  • Source DB Character Set 점검 쿼리
SELECT character_set_name
FROM information_schema.TABLES T, information_schema.COLLATION_CHARACTER_SET_APPLICABILITY CCSA
WHERE CCSA.collation_name = T.table_collation AND TABLE_SCHEMA NOT IN ( 'information_schema', 'mysql', 'performance_schema', 'sys' ) AND CCSA.character_set_name NOT IN ( 'utf8', 'utf8mb4', 'euckr' );

SELECT DEFAULT_CHARACTER_SET_NAME
FROM information_schema.SCHEMATA T
WHERE SCHEMA_NAME NOT IN ( 'information_schema', 'mysql', 'performance_schema', 'sys' ) AND DEFAULT_CHARACTER_SET_NAME NOT IN ( 'utf8', 'utf8mb4', 'euckr');
  • Source DB Collation 점검 쿼리
SELECT DEFINER FROM information_schema.ROUTINES
WHERE ROUTINE_SCHEMA NOT IN ( 'information_schema', 'mysql', 'performance_schema', 'sys' ) AND SECURITY_TYPE = 'DEFINER';

SELECT DEFINER FROM information_schema.VIEWS
WHERE table_schema NOT IN ( 'information_schema', 'mysql', 'performance_schema', 'sys' ) AND SECURITY_TYPE = 'DEFINER' ;

※ 사용자가 생성한 모든 데이터 베이스 및 테이블에 대하여 캐릭터셋을 점검 및 변경합니다.
  • 데이터 베이스 캐릭터셋 변경 예시
ALTER DATABASE [데이터베이스명] CHARACTER SET utf8 COLLATE utf8_general_ci;
  • 테이블 캐릭터셋 변경 예시
ALTER TABLE [테이블명] CONVERT TO CHARACTER SET utf8 COLLATE utf8_general_ci;

4. TargetDB가 8.0 이상 버전이고 SourceDB가 서로 다른 버전일 경우

  • MySQL 5.7 버전 및 MariaDB 10.1.7 이상 버전에서 Default 값으로 지원하는 sql_mode=’NO_AUTO_CREATE_USER’ 설정은 MySQL 8.0 버전부터는 지원하지 않습니다.
  • sql_mode 설정 점검 쿼리
SELECT ROUTINE_SCHEMA, ROUTINE_NAME, SQL_MODE
FROM information_schema.routines
WHERE ROUTINE_SCHEMA NOT IN ('sys','mysql');

SELECT TRIGGER_SCHEMA, TRIGGER_NAME, sql_mode
FROM information_schema.triggers
WHERE TRIGGER_SCHEMA NOT IN ('sys','mysql');

SELECT @@sql_mode ;
  • sql_mode 호환성 이슈 조치 방법
- Source DB > [DB Config 관리] 에서 sql_mode='NO_AUTO_CREATE_USER'  제거 설정

- Source DB > Procedure, Function, Trigger 를 삭제 후 재생성

1. Procedure, Function, Trigger 에 대해서만 drop 및 create 구문이 생성된 sql 파일 생성
   $ mysqldump -u {사용자명} -p -h {Source DB 호스트명} --set-gtid-purged=OFF --routines --triggers  --no-create-info --no-data --no-create-db --add-drop-trigger {사용자 DB} > backup.sql

2. backup.sql 파일내 NO_AUTO_CREATE_USER 구문 모두 제거
   ex) SET sql_mode = 'STRICT_TRANS_TABLES,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION'  => SET sql_mode = 'STRICT_TRANS_TABLES,NO_ENGINE_SUBSTITUTION'

3. 적용하기
   $ mysql -u {사용자명} -p -h {Source DB 호스트명} {사용자 DB} < backup.sql

5. Source DB 추가 설정

  • “log_bin = ON”, “sql_error_log_rotate = OFF” 설정합니다.
[mysqld]
server-id = 1
log-bin = binlog
sql_error_log_rotate = OFF
  • 프로시저 내 sql_error_log_rotate 변수값을 지정하는 구문 이 존재한다면 수정 또는 삭제합니다.
# 데이터 베이스에 접속하여 프로시저나 함수를 생성한 테이터베이스 선택

use [데이터베이스명];
SET sql_error_log_rotate = OFF;

네이버 클라우드 콘솔 작업 리스트

1. Source DB 정보를 바탕으로 Endpoint Management 생성

2. Migration Management에서 Migration 작업생성

  • Migration 작업 이름 지정 후 Test Connection 수행하여 이상 유무를 확인합니다.

※ 환경에 따라 Test Connection 과정에서 짧은 타임아웃 문제로 오류메시지가 발생하지 않는데, Migration 작업 시작버튼이 활성화 되지도 않는 문제가 발생할 수 있습니다. 이때는 수차례~수십차례 커넥션을 시도하면 버튼이 활성화 되기도 합니다. 그러나 지속적으로 문제가 발생할 경우에는 네이버 클라우드로 문의를 접수하여 문제를 해결해야 합니다.
  • 이상이 없을 경우 아래 “Migration 작업시작” 버튼이 활성화 됩니다. 작업시작을 클릭하면 마이그레이션을 시작합니다.

3. 마이그레이션 작업이 시작되면 Exporting, Importing, Replication과정에 에러 발생 유무를 확인합니다.

  • 작업 시작과 동시에 Cloud DB for Mysql은 운영중에서 마이그레이션 상태로 전환되며, 초기 1분이내 순단이 발생할 수 있습니다. 이후로는 정상적이 데이터 베이스 이용이 가능합니다.

  • Exporting, Importing, Replication 항목이 모두 작업이 완료될때까지 대기합니다.

  • 에러 발생 없이 작업이 완료 되면 상단에 “완료” 버튼을 클릭하여 작업을 종료합니다.
※ 마이그레이션 작업 완료 후 상단 완료 버튼을 누르기 전까지는 Source DB에 변동 데이터를 실시간으로 동기화하는 상태가 유지되며, Cloud DB for Mysql의 상태 또한 마이그레이션 상태로 유지됩니다.

※ 마이그레이션 도중 발생한 오류로 작업이 중단된 경우 재시작 버튼으로 작업을 재시작 하지 않고, 중단된 작업을 삭제 후 새로운 작업을 생성합니다. 이때 반드시 Cloud DB for MySQL에서 기본 DB를 제외한 추가된 DB를 제거 후 작업을 진행합니다.

  • 마지막으로 Target DB에 접속하여 데이터 이상 유무를 확인합니다.
